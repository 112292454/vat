# Qwen3-Omni Captioner 探索与评估记录

## 概述 (TL;DR)

本文档记录了在 VAT (Video-Auto-Translator) 项目中探索引入 `Qwen/Qwen3-Omni-30B-A3B-Captioner` 作为翻译辅助模块的完整评估过程。

**当前核心结论：**
* **暂不采用**：由于事实提取不稳定、计算开销大且与当前批处理逻辑不匹配，不建议将其集成到 VAT 的主翻译链路中。
* **定位为探索储备**：这是一个已验证完毕的探索方向。虽然未能直接落地，但在提取“情绪/场景”等弱标签方面展示了一定潜力。
* **未来的唯一可用路线**：若未来重启该路线，仅建议将其作为**旁路弱摘要器**（提取非关键的场景 Metadata），而非直接提供翻译上下文。

记录本文档的目的在于保留部署、测试与排坑经验，以供后续开发者参考，避免重复踩坑。

---

## 1. 研究动机与模型定位

引入该模型的初衷并非替换现有的翻译链，而是希望验证本地是否存在一个可用的补充模块，通过解析音频来为翻译提供额外的上下文判断（例如：识别当前是开场白、游戏高压段落、还是结束语），从而辅助系统调整翻译风格。

**`Qwen3-Omni-30B-A3B-Captioner` 的真实能力边界：**
* **它是纯音频模型（Audio-only）**：不接受文本 Prompt，输出长篇自然语言音频描述。
* **它的优势**：适合作为音频级语义摘要器、声音场景描述器，或获取口播风格、情绪、活动类型的“弱标签”。
* **它的劣势（误区避坑）**：它**不是**视觉视频 Caption 模型，**无法**作为可靠的事实提取器或专有名词识别器，**绝不能**直接将其输出用作翻译 Prompt 的强上下文。

## 2. 评估方案与发现

### 2.1 架构不匹配问题
VAT 当前默认的翻译批次是 100 条字幕（时间跨度通常在 5-14 分钟）。而 Captioner 官方建议的输入窗口必须控制在 **30 秒以内**。这意味着我们无法直接将整段音频送入模型，如果强行接入，VAT 需要额外开发一整套复杂的音频切片、窗口选择和特征压缩子系统。

### 2.2 扩展实验矩阵
为了全面评估，我们构建了以下实验矩阵：
* **样本规模**：覆盖 9 类视频内容，抽取 12 个视频的头、中、尾片段，总计 72 个样本（统一采用 30 秒窗口）。
* **窗口策略**：测试了片段起始（`start`）和中间（`midpoint`）两种截取策略。

### 2.3 运行成本极高
* **耗时统计**：单样本平均推理耗时达 149.39s（最高 255.73s），72 个样本总计耗时约 3 小时（即，几十分钟的音频要几个小时处理，开销为30G以上显存与4090显卡的算力）。
* **性能瓶颈**：窗口截取策略对耗时影响不大，真正的影响因子是**内容类型**。杂谈开场、美食实况等内容极易触发模型长篇扩写，导致推理极速变慢。

### 2.4 输出稳定性缺陷（关键劝退点）
* **BGM 系统性偏置（高发）**：高达 76.39% 的测试用例呈现音乐主导式描述。只要窗口内有明显的 BGM，模型极易产生“这是一段电子音乐/游戏配乐”的幻觉，并基于此错误前提疯狂脑补后续上下文。
* **重复与模板化塌缩**：存在连续同字符重复（占比约 4.17%）的不可用结果；并且在某些视频中，即使切换了时间窗口，模型依然会塌缩成几乎完全一致的模板化长文。

## 3. 场景表现细分

在不同的内容场景下，该模型的表现呈现出明显的两极分化：

| 场景类型 | 表现评价 | 具体表现与缺陷 |
| :--- | :--- | :--- |
| **单人游戏实况** | 🟢 相对有帮助 | 能精准捕捉紧张局面、受伤/成功反应及实况感。尽管专有名词常错，但“高强度实况”的氛围标签通常准确。 |
| **恐怖/叙事** | 🟢 相对有帮助 | 能较好地识别出“案情说明 / 悬疑叙事 / 观察异常”等偏氛围类的行为动作。 |
| **开箱/食评** | 🟡 帮助有限 | 能识别出展示动作或味道讨论，但极易被片头音乐带偏，且极其喜欢**凭空捏造**角色名、作品名和不存在的细节。 |
| **杂谈/赛车** | 🔴 严重误导 | 杂谈能抓到互动感，但会乱造人设与背景；赛车的高强度音频极易被错认为格斗游戏，引发长串重复。初配信等内容更是会在不同身份设定间反复横跳。 |

## 4. 复现与本地部署指南

如果未来出于其他目的需要重新验证，请参考以下指南：

### 4.1 环境与依赖要求
建议使用独立的虚拟环境（避免污染 VAT 主环境），需配备至少一张高显存 NVIDIA GPU（推荐多卡）。
核心依赖：`torch 2.5.1+cu121`, `transformers 4.57.3`, `accelerate 1.13.0`, `qwen-omni-utils 0.0.9`, `decord 0.6.0`。

### 4.2 最小运行示例
评估脚本依赖 VAT 已处理出的音频产物（如 `.wav` 及对应视频的目录结构）。

**跑单样本测试：**
```bash
CUDA_VISIBLE_DEVICES=0,1 USE_TF=0 TRANSFORMERS_NO_TF=1 \
  python ./scripts/qwen3_omni_caption_eval.py \
  --model-path /path/to/Qwen3-Omni-30B-A3B-Captioner \
  --output-dir /path/to/vat/output_dir \
  --database-path /path/to/vat/database.db \
  --video-id VIDEO_ID \
  --clip-duration-sec 30 \
  --work-dir /tmp/qwen3_omni_caption_eval \
  --clean-work-dir
```

**结合 Manifest 批量测试：**
```bash
CUDA_VISIBLE_DEVICES=0,1 USE_TF=0 TRANSFORMERS_NO_TF=1 \
  python ./scripts/qwen3_omni_caption_eval.py \
  --model-path /path/to/Qwen3-Omni-30B-A3B-Captioner \
  --output-dir /path/to/vat/output_dir \
  --database-path /path/to/vat/database.db \
  --manifest-json /path/to/manifest.json \
  --clip-duration-sec 30 \
  --window-strategy start \
  --window-strategy midpoint \
  --work-dir /tmp/qwen3_omni_caption_eval_manifest \
  --resume
```
运行完成后，将在 `--work-dir` 下生成 `results.json` 与 `results.md`。

## 5. 对未来的建议

鉴于该模型缺乏视觉能力、事实层极不稳定、且接入成本远大于收益，当前**果断弃用**是最合理的工程选择。

如果未来随着模型迭代或需求变化，必须要重新启用音频感知能力，建议**严格遵守以下“旁路弱摘要器”落地路线**：

1. **绝对隔离 Prompt**：绝不能将原始的长篇英文说明注入主翻译上下文。
2. **结构化信息抽取**：只提取弱标签（如 `audio_type`, `speaker_style`, `mood`, `bgm_heavy`）。
3. **强制异常过滤**：必须前置异常检测模块，拦截模板化重复、音乐主导式输出和超长文本。
4. **仅限 Metadata 辅助**：提取的数据仅写入旁路（如 `metadata['caption_context']`），现阶段只用于 Web UI 展示、Debug 面板和人工审阅。

只有在上述流程跑通且被证明“低误导、高稳定”后，才值得考虑将其重新接驳入核心翻译链路。
